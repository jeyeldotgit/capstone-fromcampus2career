# Spec 02-BJ ‚Äî `P1-S02-BJ-db-schema-app-events`

**Spec name**
`P1-S02-BJ-db-schema-app-events`

**Why this backjob exists**
The original `P1-S02-db-schema-pipeline-ops` was merged without the `app_events` outbox table.
The updated spec requires `app_events` to be in scope alongside `pipeline_jobs` and
`pipeline_rejected_rows`. This backjob adds only the missing table, constraints, indexes,
and tests. It does not reopen or re-merge any previously delivered artifact.

**Responsibility**
Add the `app_events` durable outbox table to the pipeline-ops schema layer so that internal
workflow events emitted by the TypeScript API and the Python pipeline can be persisted,
traced, and consumed reliably.

**Depends on**

- `P1-S01-db-schema-taxonomy` ‚Äî must be merged
- `P1-S02-db-schema-pipeline-ops` ‚Äî must be merged (this backjob extends it)

  section 5D (Admin Dataset Ingestion Request flow referencing `app_events`)

**In scope**

- `app_events` table definition, primary key, required columns, check constraints, indexes
- Default value enforcement: `status` defaults to `'pending'`, `available_at` defaults to
  `now()`
- Status vocabulary constraint: `pending`, `processing`, `processed`, `failed`
- Required non-null columns: `event_type`, `aggregate_type`, `payload`, `available_at`,
  `status`, `created_at`
- Index on `(status, available_at)` to support outbox polling
- Index on `(aggregate_type, aggregate_id)` to support aggregate-scoped lookups
- Table-integrity tests for the four exit criteria listed below

**Out of scope**

- Any changes to `pipeline_jobs` or `pipeline_rejected_rows`
- Any changes to taxonomy schema from Spec 01
- Prepared intelligence schema
- TypeScript services that read or emit `app_events`
- Python pipeline event-emission logic
- Seed data files
- API-layer changes

**Implementation requirements**

- `app_events.id` is a UUID primary key generated by default
- `app_events.status` has a check constraint limiting values to
  `'pending'`, `'processing'`, `'processed'`, `'failed'`
- `app_events.payload` is `jsonb not null`
- `app_events.available_at` is `timestamptz not null default now()`
- `app_events.processed_at` is nullable `timestamptz`
- `app_events.error_message` is nullable `text`
- `app_events.created_at` is `timestamptz not null default now()`
- Migration is forward-only; no rollback block required
- Drizzle schema export in `pipeline.ts` must use Drizzle ORM; no raw SQL in schema file
- Migration file must be plain SQL and must not contain TypeScript

**Decisions already made ‚Äî do not reopen**

- `app_events` is an outbox table only; it is not a Supabase Realtime subscription target
- Status vocabulary is `pending / processing / processed / failed` (not `pending / running /
complete` which belongs to `pipeline_jobs`)
- `aggregate_id` is nullable UUID to allow system-level events with no single aggregate
- `pgvector` is deferred and not relevant here

**Exit criteria (verifiable done condition)**

1. Migration applies cleanly on top of the existing `P1-S02` migration with no errors.
2. Tests verify that inserting a row without `event_type`, `aggregate_type`, `payload`, or
   `available_at` is rejected by the database.
3. Tests verify that `status` defaults to `'pending'` and that inserting a value outside the
   allowed vocabulary is rejected.
4. Tests verify that `processed_at` and `error_message` are nullable and accept null without
   error.
5. No changes are introduced to `pipeline_jobs`, `pipeline_rejected_rows`, taxonomy tables,
   or any file outside the OUTPUT list.
